Procesos ETL con GDAL
¶

GDAL (Geospatial Data Abstraction Library) es una biblioteca de software ampliamente utilizada para trabajar con datos geoespaciales. Aunque GDAL no est diseada especficamente como una herramienta de ETL (Extract, Transform, Load), puede desempear un papel importante en los procesos de ETL relacionados con datos geoespaciales al combinarse con otras herramientas.

En este tutorial, nos centraremos en los procesos ETL relacionados con datos vectoriales y utilizaremos la biblioteca OGR, que forma parte de GDAL. Esta biblioteca proporciona una interfaz sencilla para leer, escribir y manipular datos en formatos vectoriales como Shapefile, GeoPackage, GeoJSON y KML, entre otros. Utilizaremos OGR en conjunto con Python para realizar diversas tareas de transformacin y carga de datos geoespaciales vectoriales en un proceso de ETL.

Alcance¶

El propsito de este tutorial es ofrecer un enfoque prctico para llevar a cabo los procesos de transformacin y carga de diversos formatos de datos vectoriales hacia una base de datos PostGIS. Es importante destacar que no se abordarn procesos de extraccin, ya que OGR no es la herramienta mas adecuada para este propsito.

Aunque el tutorial tiene un enfoque introductorio, su objetivo es explicar de manera efectiva el uso eficiente de la biblioteca OGR para leer, filtrar y manipular datos vectoriales. Adems, aunque no sea el objetivo principal de este tutorial, tambin se incluir informacin sobre cmo establecer una conexin con una base de datos PostGIS utilizando SQLAlchemy y geopandas, as como la visualizacin de datos vectoriales desde Python a travs de las bibliotecas matplotlib y seaborn.

Requisitos previos¶

Para el desarrollo del siguiente tutorial se requiere contar con GDAL instalado y configurado en la variables de entorno de su sistema operativo.

Tener PostgreSQL instalado y configurado en la variables de entorno de su sistema operativo. Adems, debe tener el permiso necesario para crear una base de datos.

NOTA: Si bien el tutorial se desarrolla utilizando Jupyter Notebook, la parte de OGR se puede realizar sin problemas utilizando el Smbolo del sistema en Windows o el terminal en Linux, y los resultados pueden visualizarse en QGIS.

Empecemos¶

1. Crear base de datos espacial¶

Los resultados del proceso ETL deben almacenarse en una base de datos PostgreSQL con el soporte espacial de PostGIS. Por este motivo, es fundamental crear una base de datos que sea capaz de almacenar nuestros datos espaciales de manera adecuada.

La creacin de la base de datos se llevar a cabo desde el smbolo del sistema en Windows o desde la terminal en Linux. Para ello, es necesario abrir el terminal y ejecutar los siguientes comandos necesarios para crear nuestra base de datos de nombre etl y proporcionarle soporte espacial.

-- Crear base de datos "etl"
> CREATEDB -U postgres etl

-- Crear extensin PostGIS
> psql -U postgres -d etl -c "CREATE EXTENSION postgis"

-- Verificar la version de PostGIS Instalada
> psql -U postgres -d etl -c "SELECT postgis_version()"

-- Crear esquema "datos" para almacenar las capas
> psql -U postgres -d etl -c "CREATE SCHEMA datos"

2. Exportar datos vectoriales a PostGIS¶

La combinacin de OGR y PostGIS es una solucin poderosa para llevar a cabo migraciones de archivos vectoriales hacia una base de datos espacial. OGR se encarga de leer y convertir datos vectoriales en diversos formatos (como Shapefile, GeoJSON, KML, entre otros), mientras que PostGIS proporciona un almacenamiento eficiente de estos datos en tablas geoespaciales, lo que permite realizar consultas y anlisis espaciales avanzados.

A continuacin, se muestran algunos casos prcticos de cmo exportar datos vectoriales a PostGIS utilizando OGR:

Exportar archivo csv a PostGIS con ogr2ogr

Para la exportacin de datos vectoriales utilizaremos el comando ogr2ogr, una herramienta poderosa que nos permite realizar conversiones entre diferentes formatos de archivos vectoriales. Con ogr2ogr, podemos leer datos en formatos como Shapefile, GeoJSON, KML, entre otros, y exportarlos a formatos deseados o a una base de datos espacial como PostGIS.

El siguiente comando nos permitir exportar el archivo CSV con coordenadas a una base de datos espacial PostGIS

In [2]:
!ogr2ogr -f PostgreSQL -a_srs EPSG:4326 \
PG:"host=localhost dbname=etl user=postgres password=postgres" \
../datos/USLRC20230721155300.csv -where "Departamento='LIMA'" -nln eess_lima \
-OO X_POSSIBLE_NAMES=ESTE* -OO Y_POSSIBLE_NAMES=NORTE* \
-lco SCHEMA=datos -lco GEOMETRY_NAME=geom \
-lco SPATIAL_INDEX=GIST -lco OVERWRITE=YES

El comando utilizado realiza la exportacin de un archivo CSV con coordenadas a una tabla en una base de datos espacial PostGIS. Vamos a analizar cada parte del comando:

  • ogr2ogr: Es el comando principal de la herramienta GDAL que se utiliza para realizar operaciones de conversin de datos espaciales.
  • -f PostgreSQL: Especifica el formato de salida, en este caso, la base de datos PostgreSQL.
  • -a_srs EPSG:4326: Define el sistema de referencia espacial de destino para los datos, en este caso, EPSG:4326, que corresponde a WGS 84 (latitud y longitud).
  • PG:"host=localhost dbname=etl user=postgres password=postgres": Es la cadena de conexin a la base de datos PostgreSQL donde se cargarn los datos. En este caso, el comando se conectar a una base de datos llamada "etl" en el servidor local (localhost) usando el nombre de usuario "postgres" y la contrasea "postgres".
  • ../datos/USLRC20230721155300.csv: Es la ruta del archivo de origen que contiene los datos que se cargarn en la base de datos. En este caso, es un archivo CSV ubicado en la carpeta "../datos" (una carpeta anterior al directorio desde donde se ejecuta el comando).
  • -where "Departamento='LIMA'": Es una clusula SQL que se utiliza para filtrar los datos del archivo de origen antes de cargarlos en la base de datos. En este caso, solo se cargarn los registros donde el valor del campo "Departamento" sea igual a "LIMA".
  • -nln eess_lima: Especifica el nombre de la nueva capa (tabla) que se crear en la base de datos PostgreSQL para almacenar los datos. En este caso, la capa se llamar "eess_lima".
  • -OO X_POSSIBLE_NAMES=ESTE* y -OO Y_POSSIBLE_NAMES=NORTE: Estos son parmetros adicionales pasados a la capa de destino para especificar los nombres de campo en el archivo CSV que se utilizarn como coordenadas X e Y (este y norte, respectivamente). El uso del asterisco (*) indica que los nombres pueden tener algn sufijo despus de "ESTE" o "NORTE".
  • -lco SCHEMA=datos: Especifica el esquema en la base de datos PostgreSQL donde se crear la nueva capa. En este caso, la capa se crear en el esquema llamado "datos".
  • -lco GEOMETRY_NAME=geom: Define el nombre del campo que se utilizar para almacenar la geometra (coordenadas) en la capa de destino. En este caso, se utilizar un campo llamado "geom".
  • -lco SPATIAL_INDEX=GIST: Indica que se debe crear un ndice espacial GIST en el campo de geometra de la capa de destino, lo que mejorar el rendimiento de las consultas espaciales.
  • -lco OVERWRITE=YES: Este parmetro permite sobrescribir la capa de destino si ya existe en la base de datos. Si no se proporciona este parmetro y la capa ya existe, la operacin fallar.

Si el proceso se ejecuta sin errores, esto indica que la exportacin del CSV a PostGIS ha finalizado de manera satisfactoria. La visualizacin de los datos se llevar a cabo en una etapa posterior.

Exportar archivo shapefile a PostGIS con ogr2ogr

Para exportar la capa de distrito, procederemos a seleccionar los campos IDDPTO, DEPARTAMEN, IDPROV, PROVINCIA, IDDIST, DISTRITO y CAPITAL. nicamente exportaremos los registros que pertenecen al departamento de LIMA.

El comando utilizado es el siguiente:

In [4]:
!ogr2ogr -f PostgreSQL -a_srs EPSG:4326 \
PG:"host=localhost dbname=etl user=postgres password=postgres" \
../datos/DISTRITOS.shp -nlt MULTIPOLYGON -nln distritos_lima \
-sql "SELECT IDDPTO, DEPARTAMEN, IDPROV, PROVINCIA, \
             IDDIST, DISTRITO, CAPITAL \
      FROM distritos WHERE DEPARTAMEN='LIMA'" \
-lco SCHEMA=datos -lco GEOMETRY_NAME=geom \
-lco SPATIAL_INDEX=GIST -lco OVERWRITE=YES

Como se observa, este comando es similar al anterior en muchos aspectos. Ahora, nos enfocaremos en explicar los nuevos parmetros que se han agregado:

  • ../datos/DISTRITOS.shp: Es la ruta del archivo shapefile (datos/DISTRITOS.shp) que contiene los datos que se exportarn.
  • -nlt MULTIPOLYGON: Especifica el tipo de geometra de salida para las entidades. En este caso, se est configurando para que las geometras se exporten como polgonos multiparte.
  • -sql "SELECT IDDPTO, DEPARTAMEN, IDPROV, PROVINCIA, IDDIST, DISTRITO, CAPITAL FROM distritos WHERE DEPARTAMEN='LIMA'": Permite realizar una consulta SQL para filtrar los datos antes de la exportacin. Aqu, se seleccionan solo los campos especificados de la capa de distritos donde el campo DEPARTAMEN tenga el valor 'LIMA'.

Si el proceso se ejecuta sin errores, significa que la exportacin del Shapefile a PostGIS ha concluido exitosamente.

Exportar WFS a PostGIS con ogr2ogr

Para exportar la capa de sismos del WFS y filtrar nicamente aquellos registros que contengan la palabra "LIMA" en el campo "referencia", utilizaremos el siguiente comando:

!ogr2ogr -f PostgreSQL -a_srs EPSG:4326 \
PG:"host=localhost dbname=etl user=postgres password=postgres" \
WFS:"https://ide.igp.gob.pe/geoserver/CTS_sismo2020/wfs?service=WFS&version=2.0.0&request=GetCapabilities" \
-nln sismos_lima -where "referencia ILIKE '% LIMA%'" \
-lco SCHEMA=datos -lco GEOMETRY_NAME=geom \
-lco SPATIAL_INDEX=GIST -lco OVERWRITE=YES

El comando comparte similitudes con los utilizados anteriormente. Ahora, vamos a explicar los nuevos parmetros que se incluyen:

  • WFS:"https://ide.igp.gob.pe/geoserver/CTS_sismo2020/wfs?service=WFS&version=2.0.0&request=GetCapabilities": Es la URL del servicio WFS desde donde se leern los datos. El comando acceder a esta URL para obtener las caractersticas disponibles y descargar la informacin correspondiente.
  • -where "referencia ILIKE '% LIMA%'": Es una clusula de filtrado que especifica que solo se exportarn los registros donde el campo "referencia" contiene la palabra " LIMA" (la comparacin es insensible a maysculas y minsculas).

Si el proceso se ejecuta sin errores, significa que la exportacin del WFS a PostGIS ha concluido exitosamente.
